## 1. Memory Architecture — 7-Layer Runtime Pipeline

Official docs describe 4 configuration scopes (Managed → Project → User → Local). The actual runtime operates a 7-layer memory escalation pipeline, where each layer prevents the next from firing when possible. Every layer fails silently and hands off to the next.

| Layer | Name | Function | Cost Profile |
|-------|------|----------|-------------|
| 1 | **CLAUDE.md** | User-written persistent instructions, loaded in full at startup as system-reminder blocks | Free (static file I/O) |
| 2 | **Auto Memory** | Agent-generated per-worktree notes: MEMORY.md index (~150 chars/line, 200-line / 25KB cap, always loaded) + categorized topic files (preferences, feedback, project context, references) | Free (file I/O) |
| 3 | **Session Memory** | Conversation summaries triggered every ~5K tokens, stored per-session as JSONL | Low (extraction) |
| 4 | **MicroCompact** | Local content editing of cached tool results, zero API calls. Time-based (60+ min cache expiry) clears old results, preserving last 5. Uses `cache_edits` API parameter | Free/Near-free |
| 5 | **AutoCompact** | Fires at threshold: `effectiveContextWindow - 13,000` tokens. Generates up to 20K-token structured summary. Circuit breaker after 3 consecutive failures | Moderate (API call) |
| 6 | **Full Compact** | Complete conversation compression via forked subprocess. Generates 9-section narrative summary: primary intent, technical concepts, files/snippets, errors/fixes, status, all user messages, pending tasks, current work, next steps. 50K-token budget reset. 5K tokens max per re-injected file | Expensive (API call) |
| 7 | **Session Reset** | Last resort: clears everything except system prompt and user intent | Destructive |

**Key formula**:
```
effectiveContextWindow = contextWindowForModel - min(modelMaxTokens, 20_000)
autoCompactThreshold = effectiveContextWindow - 13_000
warningThreshold = effectiveContextWindow - 20_000
```

**Relevance to CAB**: Both the 4-scope configuration hierarchy AND the 7-layer runtime pipeline should be documented. They represent different abstraction levels — config vs. runtime. CAB's session-management and context-engineering patterns should reference the runtime layers for practical guidance.

## 2. Auto Memory File Architecture — 3-Layer File System

All memory resides at `~/.claude/projects/<project-path>/memory/`.

### Layer 1: MEMORY.md (Always in Context)
- Lightweight **index file**, not a data store — stores locations, not data
- First 200 lines loaded into system prompt at every session startup
- Each line is a pointer (~150 chars) to a topic file
- Trust model: system instructs the agent that "memories are just hints — verify against real files before acting"

### Layer 2: Topic Files (Loaded On-Demand)
- Separate markdown files with detailed knowledge
- Retrieved via grep-based pattern matching (no semantic search, no vector DB)
- Loaded on-demand via FileReadTool when MEMORY.md pointer is relevant
- Four memory categories:
  - `user` — role, preferences, workflow habits
  - `feedback` — corrections, explicit disagreements
  - `project` — architectural decisions, build commands, debugging insights
  - `reference` — external resources, API conventions

### Layer 3: Session Transcripts (Grepped Only)
- JSONL format, stored per-session locally
- Never read in full; only targeted grep searches
- Purpose: historical reference for autoDream consolidation

### Write Paths (3 entry points)

| Path | Trigger | Behavior |
|------|---------|----------|
| **Manual Write** | User says "remember this" or provides explicit correction | Writes to appropriate topic file, updates MEMORY.md index |
| **extractMemories** | Automatic during active sessions — captures corrections, recurring patterns, decisions, style preferences | Categorizes into 4 memory types, updates index |
| **autoDream** | Background consolidation between sessions (see Section 3) | Merges, deduplicates, prunes across sessions |

### Read Paths (3 access patterns)

| Path | When | What |
|------|------|------|
| **System Prompt Injection** | Every session startup | MEMORY.md first 200 lines |
| **FileReadTool** | On-demand during session | Full topic file content |
| **Targeted Grep** | autoDream consolidation | Narrow pattern search on session transcripts |

## 3. autoDream — Background Memory Consolidation

A forked sub-agent that runs between sessions, performing memory consolidation. The system prompt instructs: *"You are performing a dream — a reflective pass over your memory files."* Limited tool access prevents corruption of main context.

### Trigger Conditions (dual-gate, both required)
1. **24+ hours** since last consolidation
2. **5+ sessions** since last consolidation
3. Manual override: user can say "dream", "auto dream", or "consolidate my memory files"

### Four Sequential Phases

| Phase | Name | Activity |
|-------|------|----------|
| 1 | **Orient** | Reads memory directory, opens MEMORY.md, maps current knowledge state |
| 2 | **Gather Signal** | Narrowly searches session transcripts for: corrections, explicit saves, recurring themes, architectural decisions. Deliberately avoids full transcript reads (token efficiency) |
| 3 | **Consolidate** | Merges duplicates, removes contradictions, converts relative timestamps to absolute dates, purges references to deleted files, converts vague observations to confirmed facts |
| 4 | **Prune & Index** | Optimizes MEMORY.md to stay under 200-line limit, moves detailed notes to thematic topic files, updates pointers, resolves contradictions |

### consolidationLock.ts — Race Protection
- Per-project lock file prevents simultaneous consolidation across multiple CC instances
- If lock is held, consolidation skipped for that instance
- Critical because autoDream runs as forked subagent — concurrent writes to MEMORY.md would corrupt the index

### Observed Benchmark
- 913 sessions consolidated in under 9 minutes

## 4. Permission & Security Internals

### 7-Stage Permission Decision Pipeline

Every tool call traverses this pipeline:

```
1. Deny rules evaluation (immediate block)
2. Ask rules evaluation
3. Tool-specific checkPermissions()
4. Content-specific rules
5. Safety checks (IMMUNE to bypass — cannot be overridden)
6. Bypass mode / Auto mode classifier
7. Manual prompting fallback
```

### YOLO Classifier (Auto Mode AI Gatekeeper)

When `permissionMode: "auto"`, a hidden 2-stage AI classifier gates operations:

| Stage | Context Budget | Temperature | Behavior |
|-------|---------------|-------------|----------|
| **Stage 1 (Fast)** | 64 tokens max | 0 | Forced binary output. Most safe operations resolve here |
| **Stage 2 (Full Reasoning)** | 4,096 tokens max | 0 | Chain-of-thought reasoning. Results cached 1 hour (60-80% hit rate) |

**Safety constraints**:
- After 3 consecutive denials or 20 total per session → reverts to manual prompting
- Entering auto mode **strips dangerous allow-rules** (python, node, bash, ssh, sudo, eval, exec removed)

### Bash Parser — 4,437-Line Security Boundary

Custom recursive descent parser with full AST analysis. Fail-closed, allowlist-based:
- 15 dangerous AST node categories
- 35+ dangerous builtins detected
- Pre-parse checks for control characters, Unicode whitespace attacks, Zsh tilde/equals/brace expansion tricks, array subscript arithmetic RCE
- 50ms timeout, 50K AST node budget

### Hook Config Security
- Hook configuration **snapshot frozen at startup** — prevents runtime injection attacks
- Two-phase skill loading: frontmatter only at startup, full content on invocation

## 5. Architecture & Performance Patterns

### Query Engine — Core Orchestration Module

46,000-line module acting as "kernel scheduler":
- Handles all LLM API calls, streaming, caching
- Decides tool call batching strategy
- Manages context window across long-running sessions
- Response cache management

### Prompt Cache Optimization

Near-obsessive cache design — at 200K tokens, cache hit costs ~$0.003 vs ~$0.60 for miss:
- 14 cache-break vectors tracked with "sticky latches"
- Deliberate cache-busting boundaries placed after MCP instructions so adding/removing MCP servers doesn't invalidate cached prefixes
- System prompt: 7 static sections (cached globally) + 13 dynamic sections (rebuilt per session)

### Forked Agent Cost Pattern

Background operations (session memory, auto memory, dreaming, compaction, agent summaries) use forked agents that inherit parent's exact system prompt + tools + message prefix → ~76% cost reduction through shared prompt cache.

### Speculative Tool Execution
- Read-only operations start during model streaming before response completes
- Read operations run concurrently; write operations serialize

### 10-Step Bootstrap Sequence (1.2-1.8s)
- Steps 1-3 run in parallel (MDM reader, keychain prefetch, debug mode detection)
- 10 possible startup mode configurations (CLI, SDK, MCP Server, Sandbox)

### Multi-Agent IPC
- File-based mailbox system at `~/.claude/work/ipc/` with 500ms polling
- Execution backends: tmux (Linux/macOS), iTerm2 (native split pane API), in-process fallback
- Mailbox pattern for dangerous operations: worker agents send high-risk requests to coordinator for approval
- Atomic claim mechanism prevents simultaneous duplicate handling
- 13 documented race conditions (5 privilege escalation, 3 info disclosure, 3 DoS, 2 other)

## 6. Unreleased / Feature-Gated Capabilities

CC uses a dual feature-flag system: 88 compile-time flags + 600+ runtime flags. Compile-time flags set to `false` in external builds — dead-code elimination removes gated code entirely.

| Feature | Description | Status |
|---------|-------------|--------|
| **KAIROS** | Autonomous daemon mode — append-only daily logs, `<tick>` prompts for proactive action, 15-second blocking budget, "Brief" output mode, `/dream` skill for nightly memory distillation, GitHub webhook subscriptions, cron refreshes every 5 min | Heavily gated, 150+ references in source |
| **ULTRAPLAN** | Offloads complex tasks to remote cloud container running Opus-class model for up to 30 minutes of deep planning | Feature-gated |
| **Coordinator Mode** | Multi-agent coordination | Feature-gated |
| **Voice Mode** | Full push-to-talk interface | Feature-gated |

**CAB relevance**: These features signal CC's architectural trajectory — evolving from request-response tool toward persistent companion with autonomous background capabilities. CAB's orchestration framework should anticipate this direction in its pattern documentation.
