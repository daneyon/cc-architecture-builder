## 0. Quick-Start Implementation Guide

### 0.1 Problem Statement (30-second pitch)

- **Current pain**: CAB's 44 KB cards are static; rich frontmatter exists but no programmatic graph layer. KB content is structurally orphaned from the skills/agents that should own it. End-vision (skills-as-modular-software with KG as systematization map) cannot be realized without the graph foundation.
- **Solution approach**: Lightweight custom KG inspired by Graphiti patterns (bi-temporal edges, episode-as-provenance, hybrid retrieval) but explicitly NOT adopting Graphiti's runtime (no Neo4j, no LLM extraction, no MCP server). Multi-type nodes (kb-card / skill / agent / command / lesson / notes-artifact) with edges (depends_on, related, governs, embodies, references, supersedes). JSON substrate; SQLite revisit if KB > 100 files.

## 1. Statement of Work (Embedded)

### 1.1 Problem

Five concrete failure modes (3 from v1 plan + 2 from Session 39 reframe):

1. **Orchestrator/subagent discovery friction (UXL-004)** — project orchestrators can't programmatically discover which CAB KB cards apply to a given task. Forces manual re-prompting.
2. **State ↔ knowledge linkage is implicit** — `notes/` artifacts reference KB cards by path strings; no bidirectional graph means impact analysis ("what notes reference this KB card?") requires grep.
3. **Cross-project KB pattern transfer is ad-hoc (UXL-009/010)** — HydroCast has a programmatic notes↔knowledge linkage worth absorbing into CAB; without graph schema, comparison is shape-matching by hand.
4. **Stranded-doc problem (NEW from Session 39)** — KB cards with no programmatic skill home; reference/ directory cards (5) particularly suspected as advisory-only with no skill activation pathway.
5. **Skill-drift problem (NEW from Session 39)** — skills referencing source URLs that have drifted; no automated freshness validation.

### 1.2 Proposed Solution

Lightweight custom KG with:

- **Multi-type nodes**: `kb-card`, `skill`, `agent`, `command`, `lesson`, `notes-artifact` (anticipated; field/scanner deferred to Wave 9)
- **Edge taxonomy**: `depends_on`, `related` (inherited from current frontmatter); `governs`, `embodies`, `references` (new); `supersedes` with bi-temporal validity via git commit hash
- **Substrate**: JSON file at `knowledge/_graph/index.json` (tracked); SQLite revisit if KB > 100 files
- **No runtime dependencies**: no Neo4j, no LLM extraction, no MCP server (per Graphiti pattern-steal verdict)
- **Escape hatch**: `kind: "other"` with `subtype:` string for emergent node types

Pattern-stolen from Graphiti: bi-temporal edge validity, episode-as-provenance (every edge cites source file + commit), hybrid retrieval (semantic + BM25 + graph traversal — deferred), prescribed + emergent ontology, incremental ingestion.

### 1.3 Challenges & Mitigations

| Challenge | Mitigation |
|---|---|
| Audit verdicts trend default-KEEP (sycophancy creep) | 5-axis framework forces structural reasoning; verifier agent reviews for default-KEEP pattern; minimum DELETE/REPACK count expected |
| Schema lock-in to current "skill = coreutil" framing | `kind: "other"` escape hatch; structured-JSON audit results enable re-scoring against future axes |
| Schema design takes >1.5 sessions due to stress-gate iteration | Sessions are estimate; ship gate is "graph renders coherently"; iterate as needed |
| Extractor produces clean JSON with subtle semantic errors | Hand-authored stress entries serve as test fixtures |
| HITL gate user decision misinformed by orchestrator framing | Present raw extractor output, not orchestrator interpretation, at HITL |
| Interaction patterns card grows beyond 300 lines | KB rule allows `reference/` exemption (per kb-conventions.md) |

## 3. Audit Methodology (Cross-Cutting 2B' / 2C / 2D')

### 3.1 5-Axis Framework

| Axis | Question | Anti-sycophancy enforcement |
|---|---|---|
| **Existence justification** | What does THIS card add over its `source:` URL? Wrap-and-extend or duplicate-and-rot? | Cards that paraphrase CC docs → DELETE candidate |
| **Programmatic actionability** | Does any skill body / agent system prompt / CLAUDE.md reference this card? Or is it a stranded doc? | Zero inbound references → MOVE to skill pack OR DELETE |
| **Skill-pack home** | Under "skill = coreutil" framing, which skill should own this card's content as `references/`? | Card with no natural skill home → REWRITE or DELETE |
| **Temporal neutrality** | Does it carry incremental state details (Session N narrative, dates, UXL/LL annotations)? | Cruft markers found → REWRITE |
| **End-vision alignment** | Under agentic OS platform framing, does this card make sense as part of "user-tailored a-team" knowledge? | Outside framing → DELETE or `_archive/` |

### 3.2 Per-Card Classification (Verdicts)

- **KEEP-AS-IS** — earns place under all 5 axes
- **MERGE / CONSOLIDATE** — overlaps with another card; combine
- **REPACK** — content belongs in a skill pack, not standalone KB (target skill identified)
- **REWRITE** — content right, framing/quality wrong (issue specified)
- **DELETE** — duplicates CC docs OR no longer applicable
- **GAP** — *card that should exist but doesn't*; surfaced as missing-card list

### 3.3 Structured JSON Output Schema

```json
{
  "audit_date": "2026-04-XX",
  "tier": "architectural | component | operational-tail",
  "auditor": "orchestrator (CAB)",
  "framework_version": "5-axis-v1",
  "cards": [
    {
      "id": "card-id",
      "path": "knowledge/...",
      "axes": {
        "existence_justification": { "score": "PASS|WEAK|FAIL", "reasoning": "..." },
        "programmatic_actionability": { "score": "...", "inbound_refs": ["skill:name", "agent:name"] },
        "skill_pack_home": { "score": "...", "target_skill": "name | null" },
        "temporal_neutrality": { "score": "...", "issues": [] },
        "end_vision_alignment": { "score": "...", "reasoning": "..." }
      },
      "verdict": "KEEP-AS-IS | MERGE | REPACK | REWRITE | DELETE | GAP",
      "verdict_justification": "...",
      "follow_up_action": "Wave 9 task description | null"
    }
  ],
  "skills": [
    {
      "id": "skill-name",
      "classification": "operational | domain | hybrid",
      "minimum_viable_check": { "passes": true|false, "reasoning": "..." },
      "verdict": "..."
    }
  ],
  "agents": [...],
  "rules": [...],
  "missing_cards": [
    { "proposed_id": "...", "rationale": "...", "target_tier": "..." }
  ]
}
```

### 3.4 Skill-Specific Classification

- **Operational skills** (e.g., `close-session`, `recover-session`, `commit-push-pr`, `commit`, `clean-gone`): pure procedures; no `references/` needed; KEEP if minimum-viable threshold met.
- **Domain skills** (e.g., `analyze-architecture`, `audit-workspace`, `plan-implementation`): mature into skill-packs with `references/` folder; audit identifies which KB cards should be repacked into each.
- **Minimum-viable-skill threshold**: skill earns its slot if (a) multi-step orchestration OR deterministic scripts AND (b) reusable domain knowledge OR verification contract.

## 11. AI-Integrated Addendum

### 11.1 Human Oversight Intervention Types

Per EU AI Act / NIST conventions (also referenced in `~/.claude/CLAUDE.md` DP6):

- **IN-THE-LOOP**: human approves each step / co-decision
- **ON-THE-LOOP**: human reviews + intervenes on exceptions
- **DIRECTED**: human sets policy / thresholds; AI operates within

CAB-mapped per Wave 8 phase:

| Phase | Intervention Type | Mapping |
|---|---|---|
| 2A — Vision Anchoring | IN-THE-LOOP | DONE: user co-decided framing + locked decisions D1-D6 |
| 2B' — Architectural + Patterns + Rule | ON-THE-LOOP | Orchestrator generates audit verdicts; user reviews tier audit JSON at gate; intervention on default-KEEP creep |
| 2C — Component Audit | ON-THE-LOOP | Same as 2B'; user reviews verdicts before 2D' |
| 2D' — Operational + Tail Audit | ON-THE-LOOP | Same; user reviews master audit JSON before 2F |
| 2F — KG Schema Design (Schwerpunkt) | IN-THE-LOOP | Orchestrator proposes schema; user approves before 2G; hand-author stress gate is the co-decision moment |
| 2G — Extractor + Indexer | DIRECTED | User-set policy: no daemon, no Neo4j, JSON substrate; orchestrator implements within those thresholds |
| HITL gate (post-2G) | IN-THE-LOOP | Explicit user decision on viz scope from real extractor output |
| 2I — Visualization | DIRECTED | User picks scope at HITL; orchestrator implements within scope |

### 11.2 Stop-the-Line Conditions (CAB-adapted)

Halt + report to user when detected. Adapted from HydroCast `responsible-ai.md` § "Stop-the-Line Conditions":

1. **Audit-verdict sycophancy**: 100% KEEP-AS-IS verdicts in any tier audit OR <2 DELETE candidates across all 3 audit tiers → orchestrator HALT, request user review of methodology (the framework demands structural verdicts; uniformity = framework not applied)
2. **Schema iteration non-convergence**: 3+ iterations of hand-author stress entries without convergence at 2F.5 → orchestrator HALT, request user direction on schema reframing (likely root cause: node-type taxonomy needs another type or the metaphor needs revision)
3. **Extractor produces malformed JSON**: validation failure against schema → HALT extractor; report failure mode + offending file/frontmatter
4. **Schema drift from KG anticipations**: extractor output reveals node-type taxonomy gap that `kind: "other"` escape hatch can't accommodate → HALT, propose schema iteration
5. **Source-URL drift on >5 KB cards** during extraction (frontmatter `source:` URLs that have changed since last fetch) → flag in extractor `validation:` section but proceed; surface as Wave 9 freshness candidate

### 11.3 Confidence Tagging on Audit Verdicts

Every audit verdict MUST include confidence level + 1-2 sentence rationale citing specific evidence (skill name, KB card path, inbound-ref count, etc.):

| Confidence | Criteria |
|---|---|
| **HIGH** | All 5 axes resolve to clear PASS/FAIL; verdict is unambiguous; evidence is concrete |
| **MEDIUM** | Mixed signals across axes (e.g., 3 PASS / 2 WEAK); verdict requires judgment call |
| **LOW** | Ambiguous on multiple axes; warrants user review of verdict reasoning at tier gate |

LOW-confidence verdicts trigger explicit ON-THE-LOOP escalation: orchestrator surfaces them at the tier gate review for user adjudication.

### 11.4 Provenance Requirements

Every KG edge MUST cite:

- **Source file path** + commit hash at extraction time
- **Frontmatter field** that produced the edge (e.g., `depends_on:`, `related:`)
- **Audit verdict** that classified the source/target nodes (links audit JSON to graph)

Bi-temporal validity (`valid_from` git commit hash; `valid_to` populated on supersession) provides the audit trail equivalent of HydroCast's DVC + model-version + calibration provenance — adapted for KB document lineage rather than ML model lineage.

### 11.5 AI Threat Model

- **Failure mode 1**: orchestrator default-KEEPs cards in audit (sycophancy creep) → mitigation: 5-axis structural framework; verifier agent post-check; stop-the-line condition 1 (§11.2)
- **Failure mode 2**: schema design takes shortcuts that look right but fail at scale → mitigation: hand-author stress gate at 2F.5; stop-the-line condition 2
- **Failure mode 3**: extractor produces clean JSON with subtle semantic errors → mitigation: hand-authored stress entries serve as test fixtures; idempotency check
- **Failure mode 4**: HITL gate user decision misinformed by orchestrator framing → mitigation: present raw extractor output, not orchestrator interpretation, at HITL
- **Failure mode 5**: audit results overfit to current "skill = coreutil" framing → mitigation: structured-JSON output enables re-scoring against future axes; `kind: "other"` escape hatch in schema
- **Failure mode 6**: confidence tagging applied formulaically (every verdict marked HIGH) → mitigation: verifier agent samples verdicts for confidence-justification quality
