## 0. Statement of Work (Embedded)

### 0.1 Problem

CAB's knowledge base (44 markdown files across 11 directories) is **static**: rich frontmatter exists (`depends_on`, `related`, `source`, `tags`, `complexity`, etc.) but no programmatic query layer. Three concrete failure modes downstream:

1. **Orchestrator/subagent discovery friction (UXL-004)**: project orchestrators can't programmatically discover which CAB KB cards apply to a given task. Forces manual re-prompting.
2. **State ↔ knowledge linkage is implicit**: `notes/` artifacts (sessions, plans, lessons) reference KB cards by path strings; no bidirectional graph means impact analysis ("what notes reference this KB card?") requires grep.
3. **Cross-project KB pattern transfer is ad-hoc (UXL-009/010)**: HydroCast has a programmatic notes↔knowledge linkage worth absorbing into CAB; without graph schema, comparison is shape-matching by hand.

### 0.2 Proposed Solution

**Five-phase architecture** producing (a) a queryable graph representation of existing KB, (b) a notes/ ↔ knowledge/ linking convention, and (c) a visualization surface for human review. The user's end-vision (KB → domain-specialized skill packs) is **served downstream** by this foundation: the Phase 3c.2 `scaffold-project` router pattern (Knowledge Anchors links) is the seed; once graph data exists, automated repacking becomes feasible (UXL-009/010 territory, post-Wave-8).

### Critical findings

- **Zero files missing frontmatter** — full graph eligibility
- **Zero files missing required `source:`** — wrapper philosophy + LL-11 fresh-fetch convention well-enforced
- **44 unique IDs / 44 files** — clean 1:1 ID-to-file mapping; no collisions
- **8 dangling cross-references** — all in `knowledge/reference/`, all pointing at `plan-implementation` or `execute-task` (which are SKILL names, not KB card IDs). This is a **cross-domain edge case**: KB cards correctly cross-reference skills, but skills aren't KB nodes in the current schema.

### Dangling cross-refs detail (8)

```
knowledge/reference/prioritization-frameworks.md: related -> plan-implementation
knowledge/reference/product-design-cycle.md:      related -> plan-implementation
knowledge/reference/product-design-cycle.md:      related -> execute-task
knowledge/reference/requirements-doc-guide.md:    related -> plan-implementation
knowledge/reference/ux-testing-agentic-os.md:     related -> plan-implementation
knowledge/reference/ux-testing-agentic-os.md:     related -> execute-task
knowledge/reference/visualization-workflow.md:    related -> plan-implementation
knowledge/reference/workflow-processflow.md:      related -> plan-implementation
```

These are NOT bugs to fix — they're a **schema requirement signal for Phase 2**: the graph must accommodate **multi-type nodes** (KB cards, skills, agents, commands) so cross-domain edges resolve cleanly. This is exactly the user's end-vision territory: KB ↔ skills ↔ agents form a unified graph.

### Implications for Phase 2 (Graph Schema Design)

1. **Multi-type nodes required**: not just KB cards. Skills + agents + commands are first-class graph nodes too. Frontmatter `name:` field on skills/agents/commands serves as their ID (just like `id:` on KB cards).
2. **Edge type taxonomy needs expansion**:
   - `depends_on` (KB → KB, conceptual prerequisite)
   - `related` (any → any, lateral association)
   - `governs` (LL → skill/agent/command/rule, structural enforcement) — NEW; emerges from `lessons-learned.md` `Operational embodiments by LL` table
   - `embodies` (skill/agent/command → LL, reverse of governs) — NEW
   - `references` (notes/ → KB or notes/ → skill, mention without semantic dependency) — NEW for Phase 4 notes/ ↔ knowledge linkage
3. **The 8 dangling refs become valid edges** once skills are nodes. No KB content changes needed.
4. **`category:` field already provides clustering** for visualization (Phase 5) — no need to design clustering separately.

### KB readiness verdict

**KB is graph-ready.** The 100% coverage on required fields + 1:1 ID mapping + clean tag taxonomy + only-cross-domain dangling refs means Phase 2 schema design can focus on **node-type expansion + edge-type taxonomy**, not on remediating bad data. This is a much cleaner starting point than expected — the audit could have found 30%+ frontmatter gaps; instead found 0%.
