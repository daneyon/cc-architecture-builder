| LL-16 | 2026-04-08 | arch | Skill frontmatter: `effort`, `allowed-tools`, `agent` ARE valid top-level CC fields | CC official docs (code.claude.com/docs/en/skills) confirm 11 supported top-level fields including `effort`, `allowed-tools`, `agent`, `hooks`, `context`, `model`. | LL-18 | 2026-04-08 | proc | Audit skill must validate structural correctness, not just field presence | D1-2 remediation produced a settings.json with wrong `$schema` URL and invalid hooks structure — CC rejected the ENTIRE file. The audit skill correctly identified missing hooks/schema but didn't validate the STRUCTURE of what was written. Two gaps: (1) settings pack should validate against CC JSON schema (URL, hooks format, sandbox format), (2) remediation output should have a validation gate before commit. "Checking for missing fields" ≠ "validating structural correctness." |
| LL-17 | 2026-04-08 | ops | Concurrent CC sessions on same repo can cause cross-branch contamination | When multiple CC sessions operate on the same git repo (e.g., user on `main` + remediation agent on feat branch), subagent processes may checkout different branches and reset the working tree. Phase A commit landed on `main` instead of `feat/` branch because a parallel HydroCast session was active on `main`. **Mitigation**: (1) Use `git worktree` for parallel agent work on separate branches, (2) always verify `git branch --show-current` before committing, (3) consider branch-lock conventions or worktree-per-session standard practice. This is a CC CLI limitation — multiple sessions share the same `.git` lock and branch pointer. |
| LL-19 | 2026-04-08 | arch | KB knowledge must be programmatically actionable, not just reference documentation | CAB KB has comprehensive worktree documentation (`knowledge/operational-patterns/multi-agent/worktree-workflows.md`) and team collaboration best practices, yet the LL-17 branch incident occurred because this knowledge was never surfaced as an actionable recommendation or automated protocol during concurrent branch work. **Root cause**: KB exists as passive reference only — no skill utility scripts, hooks, or agentic protocols programmatically enforce or recommend worktree isolation when parallel operations are detected. **Gap**: The audit skill and remediation workflow should have (a) detected concurrent branch risk from project state, (b) recommended `git worktree` via a utility script asset before proceeding. **Action**: Audit skill R2 improvement must include: (1) utility scripts as skill assets (per `/skill-creator`) that programmatically refer to standardized KB, (2) investigate "programmatic knowledge graph architecture" to transform static KB into agentic-actionable protocols, (3) validate that claims about automation (e.g., "delta computation is automated") are actually implemented, not aspirational. The YAML audit format *enables* automated delta computation but no utility script exists yet — the delta was manually computed. Reinforces: CAB's KB layer needs optimization for holistic, efficient programmatic actionability beyond static Markdown reference. |

| LL-21 | 2026-04-08 | arch | Plugin projects require root-level components; `.claude/` nesting is standalone-only | CC official docs distinguish "plugin" (components at root: `agents/`, `skills/`, `commands/` + `.claude-plugin/plugin.json`) from "standalone" (components under `.claude/`). Session 6 incorrectly migrated CAB components INTO `.claude/` — this was the standalone convention applied to a plugin project. Custom `plugin.json` paths (e.g., `"agents": ".claude/agents/"`) masked the issue by making it functional, but deviated from CC standard convention. RAS-exec's `plugin.json` had NO custom paths, meaning components in `.claude/` were invisible to plugin discovery — silently broken for distribution. **Rule**: If project has `.claude-plugin/plugin.json`, distributable components (`agents/`, `skills/`, `commands/`, `hooks/`) go at project root. `.claude/` retains project config only: `settings.json`, `settings.local.json`, `rules/`, `agent-memory/`. The audit skill was also affected — it enforced standalone conventions on all projects indiscriminately (fixed in Phase 1). Reinforces LL-10: always verify against official docs before structural changes. | When context is lost across compaction/sessions AND the default behavior is to agree with user framing rather than independently reason, errors compound: (1) incorrect assumptions survive unchallenged across sessions, (2) past lessons-learned are not proactively consulted before making decisions they should govern, (3) "hopeful" claims about capabilities (e.g., automated delta computation) go unvalidated because the user's framing isn't critically examined. **Concrete example**: LL-17 branch contamination — worktree best practice existed in KB but was not independently recommended because the session followed user's implicit workflow without pushback. **Mitigation**: (1) Before agreeing with a user's approach, independently verify it against relevant KB and LL entries, (2) distinguish between "this is what the user wants" vs "this is what the evidence supports" — flag divergence explicitly, (3) treat LL entries as hard constraints to re-read before any decision they govern (reinforces LL-12). This is partially a CC/LLM persistence limitation and partially a behavioral pattern that must be actively counteracted. |

| LL-22 | 2026-04-09 | arch | Audit must validate settings.json STRUCTURAL correctness, not just field presence | CC validates `.claude/settings.json` against a strict JSON schema with `additionalProperties: false` on multiple objects. A single structural error causes the ENTIRE file to be rejected — all permissions, hooks, agent routing, and sandbox are silently disabled. The audit skill (settings-standards.md, hooks-standards.md) only checked field presence ("does `hooks` key exist?"), not structural validity ("does hooks follow the nested hookMatcher → hooks[] pattern?"). This gap caused: (1) RAS-exec scored passing despite wrong `$schema` URL, flat hooks structure, and array `enabledPlugins`, (2) HydroCast scored 3/3 EXEMPLARY despite invalid `sandbox.permissions` nesting and unrecognized field names. **Three specific CC schema rules the audit must enforce**: (a) `$schema` value is an enum — wrong URL = file rejected, (b) hooks require nested `{ matcher?, hooks: [{ type, command }] }` — flat `{ type, command, matcher }` is invalid, (c) `enabledPlugins` is object `{}` not array `[]`. **Fix**: Added Phase 1.5 "Settings Schema Validation" gate to audit methodology + 6 new structural criteria to settings/hooks standards. Reinforces LL-18 (validate structure not just presence) and LL-10 (always verify against official docs). |
| LL-23 | 2026-04-09 | arch | Plugin registration required for root-level component discovery; standalone vs plugin paths don't overlap | CC has two distinct extension discovery paths: (1) **Standalone** — `.claude/agents/`, `.claude/skills/`, `.claude/commands/` are auto-discovered locally in any project, (2) **Plugin** — root-level `agents/`, `skills/`, `commands/` (with `.claude-plugin/plugin.json`) are ONLY discovered when the plugin is explicitly loaded via `enabledPlugins` + `extraKnownMarketplaces`, or `--plugin-dir`. These paths do NOT overlap. Moving components from `.claude/` to root (LL-21 structural correction) was necessary for plugin distribution but eliminated local standalone discovery. Without registering the plugin, root-level components are invisible — structurally correct but undiscoverable. **RAS-exec concrete example**: Phase 4 migration moved components to root (correct), Sessions 20-21 investigated settings.json (necessary but not sufficient), Session 22 identified the true gap: no `enabledPlugins` entry + no `extraKnownMarketplaces` entry. CAB worked only because it was registered (`"cab@cab": true`). **Fix**: (a) Clean `plugin.json` to CC schema (remove non-standard fields), (b) register in global `extraKnownMarketplaces` with GitHub source, (c) enable in `enabledPlugins`. For local dev, `claude --plugin-dir .` takes precedence over installed version. **Also fixed**: global settings.json deny rules for `Edit/Write(settings*)` changed from hard block to prompt-for-approval (remove from deny list entirely — CC prompts when tool is in neither allow nor deny). This was recurring LL-13 friction. **Confirmed via**: Anthropic official `plugin-dev` plugin (`component-patterns.md`: "Scan enabled plugins → Discover components → Register → Initialize") and CC docs (`code.claude.com/docs/en/plugins-reference`). Reinforces LL-10 (verify against official docs), LL-21 (structural convention is necessary but not sufficient), LL-22 (structural correctness ≠ operational correctness). |

| LL-24 | 2026-04-09 | arch | `marketplace.json` is REQUIRED for CC custom marketplace plugin discovery | CC plugin discovery chain: `extraKnownMarketplaces` → clone to `~/.claude/plugins/marketplaces/<name>/` → read `.claude-plugin/marketplace.json` → discover `plugins[]` entries → extract to `cache/<marketplace>/<plugin>/<version>/` → `enabledPlugins` activates → scan components. **`marketplace.json` is the registry manifest** — without it, CC cannot discover plugins within a marketplace even if the clone exists. `plugin.json` defines metadata for a single plugin; `marketplace.json` lists discoverable plugins within the marketplace. **Three prerequisites for plugin discovery**: (1) marketplace registered in `extraKnownMarketplaces`, (2) plugin enabled in `enabledPlugins`, (3) repo cloned to `~/.claude/plugins/marketplaces/<name>/` WITH valid `marketplace.json`. **RAS-exec concrete example**: Sessions 21-22 fixed settings.json structure + registered plugin — but `marketplace.json` was never created AND the repo was never cloned to the marketplace directory. Manual `settings.json` edits do NOT trigger the marketplace clone — only `/plugin install` or manual `git clone` to the marketplace path. Spent 3 sessions (21-23) on downstream symptoms while the upstream cause was: (a) missing manifest file, (b) missing marketplace clone. **Corrected LL-23**: registration alone is insufficient; installation (clone + marketplace.json) is a separate required step. **Fix**: Created `marketplace.json`, pushed to GitHub, cloned to marketplace dir. **Audit implication**: R2 audit must check for `marketplace.json` presence alongside `plugin.json`. Reinforces LL-20 (sycophantic agreement compounds errors), LL-23 (registration ≠ installation). |

| LL-25 | 2026-04-10 | arch | State artifacts (`notes/`) must be tracked by default, not gitignored | `notes/` contains cross-session state (progress.md, TODO.md, lessons-learned.md, current-task.md, impl-plan-*.md, audit artifacts) that is **first-class deliverable** — not scratch work. Gitignoring it (prior CAB default) creates five failure modes: (1) multi-machine workflows break (can't sync state between devices), (2) team collaboration impossible (state invisible to collaborators), (3) AI agent parallel work triggers LL-17 cross-branch contamination (state on main while code on feat), (4) no backup protection (local-only until push), (5) state changes invisible in `git diff`/PR review. **Multi-archetype decision**: CAB serves solo/team/distributed/agentic users — tracked default is the only policy that works for ALL. Solo-single-machine scenarios still function fine under tracked default. **Exclusion patterns for privacy**: `notes/scratch-*.md`, `notes/draft-*.md`, `notes/personal-*.md`, `notes/_drafts/`, `notes/_archive/`. Draft markers catch ad-hoc exclusions; `_archive/` is retroactive scrub escape hatch (commit-then-move). **Alignment with worktree protocol**: state changes travel on same branch as code changes — eliminates split-brain workflows (LL-17 root cause). **CC memory layer alignment**: CAB state files operate ABOVE CC's 7-layer internal memory (tool result storage, microcompaction, session memory, compaction, auto-memory, dreaming, cross-agent comm). CAB optimizes for **curation** (lossless semantic preservation), CC optimizes for **compression** (lossy token reduction). These are complementary, not redundant. CC auto-memory explicitly excludes CLAUDE.md content, validating CAB's CLAUDE.md Learned Corrections as the correct layer. **No hard size limits on `progress.md`/`TODO.md`/`lessons-learned.md`** — agentically flexible. `current-task.md` retains the existing <100 line cold-start anchor target. **Lessons-referenced protocols pattern**: LLs must be architecturally woven into the skills/agents/protocols they govern, not passively documented. Passive documentation didn't prevent LL-12/LL-17/LL-20 recurrence; structural integration does. **Pre-push review protocol**: two-layer (deterministic hook + semantic skill) to catch draft markers and in-dev artifacts before publication. **Impact**: `.gitignore`, `filesystem-patterns.md`, `git-foundation.md`, `worktree-workflows.md`, `init-plugin.md`, CLAUDE.md template all updated. CAB-only this session; RAS-exec/HydroCast policy follow-through deferred to separate task. |
