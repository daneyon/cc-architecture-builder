| LL-17 | 2026-04-08 | ops | Concurrent CC sessions on same repo can cause cross-branch contamination | When multiple CC sessions operate on the same git repo (e.g., user on `main` + remediation agent on feat branch), subagent processes may checkout different branches and reset the working tree. | LL-21 | 2026-04-08 | arch | Plugin projects require root-level components; `.claude/` nesting is standalone-only | CC official docs distinguish "plugin" (components at root: `agents/`, `skills/`, `commands/` + `.claude-plugin/plugin.json`) from "standalone" (components under `.claude/`). Custom `plugin.json` paths (e.g., `"agents": ".claude/agents/"`) masked the issue by making it functional, but deviated from CC standard convention. RAS-exec's `plugin.json` had NO custom paths, meaning components in `.claude/` were invisible to plugin discovery — silently broken for distribution. **Rule**: If project has `.claude-plugin/plugin.json`, distributable components (`agents/`, `skills/`, `commands/`, `hooks/`) go at project root. `.claude/` retains project config only: `settings.json`, `settings.local.json`, `rules/`, `agent-memory/`. The audit skill was also affected — it enforced standalone conventions on all projects indiscriminately (fixed in Phase 1). Reinforces LL-10: always verify against official docs before structural changes. | When context is lost across compaction/sessions AND the default behavior is to agree with user framing rather than independently reason, errors compound: (1) incorrect assumptions survive unchallenged across sessions, (2) past lessons-learned are not proactively consulted before making decisions they should govern, (3) "hopeful" claims about capabilities (e.g., automated delta computation) go unvalidated because the user's framing isn't critically examined. **Concrete example**: LL-17 branch contamination — worktree best practice existed in KB but was not independently recommended because the session followed user's implicit workflow without pushback. **Mitigation**: (1) Before agreeing with a user's approach, independently verify it against relevant KB and LL entries, (2) distinguish between "this is what the user wants" vs "this is what the evidence supports" — flag divergence explicitly, (3) treat LL entries as hard constraints to re-read before any decision they govern (reinforces LL-12). | LL-22 | 2026-04-09 | arch | Audit must validate settings.json STRUCTURAL correctness, not just field presence | CC validates `.claude/settings.json` against a strict JSON schema with `additionalProperties: false` on multiple objects. A single structural error causes the ENTIRE file to be rejected — all permissions, hooks, agent routing, and sandbox are silently disabled. The audit skill (settings-standards.md, hooks-standards.md) only checked field presence ("does `hooks` key exist?"), not structural validity ("does hooks follow the nested hookMatcher → hooks[] pattern?"). This gap caused: (1) RAS-exec scored passing despite wrong `$schema` URL, flat hooks structure, and array `enabledPlugins`, (2) HydroCast scored 3/3 EXEMPLARY despite invalid `sandbox.permissions` nesting and unrecognized field names. **Three specific CC schema rules the audit must enforce**: (a) `$schema` value is an enum — wrong URL = file rejected, (b) hooks require nested `{ matcher?, hooks: [{ type, command }] }` — flat `{ type, command, matcher }` is invalid, (c) `enabledPlugins` is object `{}` not array `[]`. **Fix**: Added Phase 1.5 "Settings Schema Validation" gate to audit methodology + 6 new structural criteria to settings/hooks standards. Reinforces LL-18 (validate structure not just presence) and LL-10 (always verify against official docs). |
| LL-23 | 2026-04-09 | arch | Plugin registration required for root-level component discovery; standalone vs plugin paths don't overlap | CC has two distinct extension discovery paths: (1) **Standalone** — `.claude/agents/`, `.claude/skills/`, `.claude/commands/` are auto-discovered locally in any project, (2) **Plugin** — root-level `agents/`, `skills/`, `commands/` (with `.claude-plugin/plugin.json`) are ONLY discovered when the plugin is explicitly loaded via `enabledPlugins` + `extraKnownMarketplaces`, or `--plugin-dir`. These paths do NOT overlap. Moving components from `.claude/` to root (LL-21 structural correction) was necessary for plugin distribution but eliminated local standalone discovery. Without registering the plugin, root-level components are invisible — structurally correct but undiscoverable. **RAS-exec concrete example**: Phase 4 migration moved components to root (correct), Sessions 20-21 investigated settings.json (necessary but not sufficient), Session 22 identified the true gap: no `enabledPlugins` entry + no `extraKnownMarketplaces` entry. CAB worked only because it was registered (`"cab@cab": true`). **Fix**: (a) Clean `plugin.json` to CC schema (remove non-standard fields), (b) register in global `extraKnownMarketplaces` with GitHub source, (c) enable in `enabledPlugins`. For local dev, `claude --plugin-dir .` takes precedence over installed version. **Also fixed**: global settings.json deny rules for `Edit/Write(settings*)` changed from hard block to prompt-for-approval (remove from deny list entirely — CC prompts when tool is in neither allow nor deny). This was recurring LL-13 friction. **Confirmed via**: Anthropic official `plugin-dev` plugin (`component-patterns.md`: "Scan enabled plugins → Discover components → Register → Initialize") and CC docs (`code.claude.com/docs/en/plugins-reference`). Reinforces LL-10 (verify against official docs), LL-21 (structural convention is necessary but not sufficient), LL-22 (structural correctness ≠ operational correctness). |

| LL-24 | 2026-04-09 | arch | `marketplace.json` is REQUIRED for CC custom marketplace plugin discovery | CC plugin discovery chain: `extraKnownMarketplaces` → clone to `~/.claude/plugins/marketplaces/<name>/` → read `.claude-plugin/marketplace.json` → discover `plugins[]` entries → extract to `cache/<marketplace>/<plugin>/<version>/` → `enabledPlugins` activates → scan components. **`marketplace.json` is the registry manifest** — without it, CC cannot discover plugins within a marketplace even if the clone exists. `plugin.json` defines metadata for a single plugin; `marketplace.json` lists discoverable plugins within the marketplace. **Three prerequisites for plugin discovery**: (1) marketplace registered in `extraKnownMarketplaces`, (2) plugin enabled in `enabledPlugins`, (3) repo cloned to `~/.claude/plugins/marketplaces/<name>/` WITH valid `marketplace.json`. **RAS-exec concrete example**: Sessions 21-22 fixed settings.json structure + registered plugin — but `marketplace.json` was never created AND the repo was never cloned to the marketplace directory. Manual `settings.json` edits do NOT trigger the marketplace clone — only `/plugin install` or manual `git clone` to the marketplace path. Spent 3 sessions (21-23) on downstream symptoms while the upstream cause was: (a) missing manifest file, (b) missing marketplace clone. **Corrected LL-23**: registration alone is insufficient; installation (clone + marketplace.json) is a separate required step. **Fix**: Created `marketplace.json`, pushed to GitHub, cloned to marketplace dir. **Audit implication**: R2 audit must check for `marketplace.json` presence alongside `plugin.json`. Reinforces LL-20 (sycophantic agreement compounds errors), LL-23 (registration ≠ installation). |

Gitignoring it (prior CAB default) creates five failure modes: (1) multi-machine workflows break (can't sync state between devices), (2) team collaboration impossible (state invisible to collaborators), (3) AI agent parallel work triggers LL-17 cross-branch contamination (state on main while code on feat), (4) no backup protection (local-only until push), (5) state changes invisible in `git diff`/PR review. **Multi-archetype decision**: CAB serves solo/team/distributed/agentic users — tracked default is the only policy that works for ALL. Draft markers catch ad-hoc exclusions; `_archive/` is retroactive scrub escape hatch (commit-then-move). **Alignment with worktree protocol**: state changes travel on same branch as code changes — eliminates split-brain workflows (LL-17 root cause). **CC memory layer alignment**: CAB state files operate ABOVE CC's 7-layer internal memory (tool result storage, microcompaction, session memory, compaction, auto-memory, dreaming, cross-agent comm). CAB optimizes for **curation** (lossless semantic preservation), CC optimizes for **compression** (lossy token reduction). These are complementary, not redundant. Passive documentation didn't prevent LL-12/LL-17/LL-20 recurrence; structural integration does. **Pre-push review protocol**: two-layer (deterministic hook + semantic skill) to catch draft markers and in-dev artifacts before publication. **Impact**: `.gitignore`, `filesystem-patterns.md`, `git-foundation.md`, `worktree-workflows.md`, `init-plugin.md`, CLAUDE.md template all updated. | LL-26 | 2026-04-10 | proc | State file tense hygiene + two-commit session close pattern | Session 24 committed work + state files together in `302f872` with state files saying `"EXECUTED ✅ — Ready for commit + session close"` — true at write time, **stale the instant the commit landed**. This is the exact failure mode LL-25's "Lessons-Referenced Protocols" pattern was created to prevent — it slipped through because LL-25 itself was the work being committed. **Root cause (four protocol gaps)**: (1) `skills/session-close/` Step 4 commits state updates but doesn't distinguish work-commit from state-refresh-commit, (2) `skills/executing-tasks/` Phase 5 says update state AFTER commit but doesn't require a second commit for the refresh nor forbid "pending" tense, (3) pre-push hook regex doesn't include tense markers, (4) no documented tense-hygiene convention in `filesystem-patterns.md`. **Corrective protocol (DD-1 two-commit pattern)**: Default session close = work-commit lands first → refresh state files using past-tense framing referencing the work-commit's hash → commit state refresh as `chore(session-NN): refresh state post-<commit>`. Token cost ~130/session (0.065% budget) = negligible. Tense-neutral single-commit is a lightweight fallback for mid-session state touches only. **Forbidden status-line patterns**: `"ready for commit"`, `"pending commit"`, `"awaiting commit"`, `"EXECUTED ✅ — ready for..."`, `"will commit"`. **Approved patterns**: `"executed in <hash>"`, `"executed 2026-XX-XX"`, `"committed in <hash>"`, `"landed in <hash>"`. **Enforcement layers (architecturally woven, not passive)**: (a) `filesystem-patterns.md` documents tense convention + two-commit protocol, (b) `skills/session-close/SKILL.md` Step 4 enforces two-phase close, (c) `skills/executing-tasks/SKILL.md` Phase 5 requires post-commit refresh + second commit, (d) pre-push hook regex catches stale status lines with **anchored status-line pattern** (`^\*\*(Status|Phase|Gate)\*\*:.*(pending commit|ready for commit|awaiting commit|will commit)`) — descriptive prose in body text, code fences, or headings is explicitly allowed. The anchoring nuance is critical: naive unanchored matching false-positives on plans/docs that reference the concept descriptively (Session 25 smoke test caught this). **Commit-per-phase cadence (DD-4)**: Recommended guidance (not prescription) added to `executing-tasks` for multi-phase tasks — clean git history for audit trails. **Folded with LL-25 follow-on**: A.5 anchored-regex approach also tightens the existing `\bWIP\b` false-positive pattern using the same status-line anchoring — resolves two issues in one hook touch, eliminates the `CAB_SKIP_PREPUSH_REVIEW=1` bypass dependency. **Smoke test**: Retroactively apply A.5 regex to Session 24's `current-task.md` — must match `**Status**: EXECUTED ✅ — Ready for commit + session close` as a stale status line. **Reinforces**: LL-25 (lessons-referenced protocols must be structurally enforced), LL-20 (sycophantic agreement + persistence gaps compound — protocol enforcement counteracts both). |

| LL-27 | 2026-04-11 | arch | Agent name-resolution precedence creates silent shadowing between global and plugin agents | CC agent resolution order is **local `.claude/agents/` → user `~/.claude/agents/` → plugin `<plugin>/agents/`**. When a plugin-authored agent (e.g., CAB's `orchestrator`) has a name collision with a file at a higher-precedence layer, the plugin version is silently shadowed — loaded in registry but never selected. **Concrete incident (Session 26 diff analysis)**: CAB's R2 remediation updated `agents/orchestrator.md` across Sessions 19-26 (LL-15 `context:` removal, LL-21 plugin-first paths, permissionMode cleanup, extension registry updates). A stale copy at `~/.claude/agents/orchestrator.md` from April 7 silently shadowed CAB's plugin version — meaning `"agent": "orchestrator"` in global `settings.json` resolved to the April-7 snapshot, not CAB's current version. All R2 remediation improvements were invisible in operational reality for ~2 weeks. Same risk applies to `verifier.md` and potentially any future name collision. **Root cause of the silent drift**: Plugin-first architecture (LL-21) + plugin registration (LL-23) + marketplace discovery (LL-24) together made CAB's orchestrator **available** but agent name-resolution precedence made it **non-default**. Availability ≠ selection. The shadow copy problem does not trigger any error, warning, or log entry — it's invisible at runtime. **Three detection paths**: (a) `/sync-check` drift detection (CAB extension — must be extended to flag name collisions, not just file absence), (b) fresh-session agent introspection (ask CC which file resolved the agent name — method TBD, may require plugin API), (c) manual diff between plugin source and any same-named file at higher precedence layer. **Corrective pattern**: For single-source-of-truth plugins like CAB, delete shadow copies from global (`~/.claude/agents/orchestrator.md`, `~/.claude/agents/verifier.md`) immediately upon plugin adoption, and document the expectation in global `CLAUDE.md` Extension Registry. **Architectural layering clarification (surfaced during the Session 26 investigation)**: Identity (persona, philosophy, domain guidelines) belongs in `~/.claude/CLAUDE.md` — this is what makes a global orchestrator YOURS and is load-bearing across all sessions. Behavior (classify/route logic, PLAN→VERIFY→COMMIT protocol, delegation heuristics) belongs in the plugin's `agents/<name>.md` — this is swappable/improvable via the plugin's versioned lifecycle. Selection (`"agent": "orchestrator"` pointer) lives in global `settings.json` — this is just a name reference, not content. Mixing identity into the behavioral file is what caused operators to resist deletion (felt like "losing my orchestrator"), when in reality identity survives deletion because it lives one layer up. **Enforcement layer additions needed**: (1) extend `/sync-check` to detect plugin↔global name collisions, (2) add name-collision section to `knowledge/operational-patterns/multi-agent/agent-resolution.md` (NEW KB card), (3) add shadow-check to `/validate --cab-audit` methodology, (4) add explicit "CAB provides this — do not create a global copy" note to orchestrator/verifier extension registry entries. **Reinforces**: LL-15 (fixes don't propagate if loaded-but-shadowed), LL-19 (KB knowledge must be programmatically actionable — resolution rules are currently implicit), LL-23/24 (plugin availability ≠ plugin precedence). |

| LL-28 | 2026-04-11 | proc | State files capture plan-level state well but miss dialogue-level emergent content when sessions die mid-dialogue | LL-26 solved **tense staleness** (state files describing "pending" commits that already landed). It does not solve **emergence staleness** — discoveries, analyses, and decisions produced during execution *after* the last state-refresh checkpoint. **Concrete incident (Session 26 death via "Prompt is too long")**: Between the Phase 5b state refresh commit (`726c50b`) and the `Prompt is too long` response at context exhaustion, the session also completed: Phase B.5 push (clean, no bypass), Phase C.1 inventory (4 command + 2 skill + 2 agent duplicates identified), Phase C.1 diffs (all CAB versions confirmed as strict supersets), **the LL-27 shadowing discovery** (session's most important finding), a 3-layer Identity/Behavioral/Selection architectural framing (new pattern), an Option A/B/C analysis with 4-lens reasoning recommending Option B + 4 safeguards, and the user's unanswered clarifying question about global↔plugin orchestrator authority. **None** of that reached `notes/` because the LL-26 protocol triggers state writes on *phase boundaries*, not on *discovery/decision events*. Session 27 had to reconstruct it from the JSONL transcript — the exact scenario tracked-notes-by-default was supposed to eliminate. **Root cause**: `executing-tasks` Phase 5 and `session-close` Step 2 both write state at the end of a phase. A session that dies mid-phase (mid-dialogue, mid-investigation, mid-HITL discussion) loses everything since the last phase-end write. **Corrective protocol (candidate, not yet implemented)**: Introduce **event-triggered state writes** alongside phase-triggered writes. Trigger events: (a) new architectural insight or pattern worth capturing, (b) HITL decision point reached (write pending question + options to state before asking user), (c) investigation finding that contradicts or updates a prior LL/DD, (d) discovery that implies a new LL entry. Implementation sketch: add a lightweight "dialogue checkpoint" step to `executing-tasks` — after any assistant turn that produces >N tokens of analytical content OR poses an HITL question, append a 5-10 line "dialogue state" block to `progress.md` before the next tool call. Cost: ~200-400 tokens per checkpoint, far cheaper than context-exhaustion recovery. **Fallback recovery protocol (for existing dying sessions)**: When `Prompt is too long` strikes before the next checkpoint, new session must (1) read `notes/` for plan state, (2) read last ~20 turns of dying-session JSONL transcript for dialogue state, (3) backfill emergent content into `notes/` as Session N+1 first action, (4) draft any new LL entries surfaced during dying session, (5) resume at the exact HITL question the dying session was blocked on. **Meta-lesson on CAB's iterative optimization**: State management standardization is itself iteratively optimized — each recurring failure mode reveals a new protocol gap. Track protocol additions via git so each layer is individually revertable. Do not hard-code any new protocol until it has passed at least one real-world dying-session recovery test. **Reinforces**: LL-25 (tracked notes are first-class deliverables), LL-26 (tense hygiene), LL-20 (context persistence gaps compound errors), LL-03 (post-compaction drops context — `Prompt is too long` is a special case of forced-but-failed compaction). |
