## Schema

| Field | Values | Meaning |
|---|---|---|
| **ID** | `LL-NN` | Stable identifier; never reused, never renumbered |
| **Date** | `YYYY-MM-DD` | Date the lesson was first recorded |
| **Cat** | `ops` / `arch` / `ctx` / `proc` / `tool` | Domain — operational, architectural, context-engineering, process, tooling |
| **Class** | `INTEGRATED` / `ACTIVE` / `ADVISORY` / `ARCHIVED` | Structural integration state (see below) |
| **Pri** | `P0` / `P1` / `P2` / `—` | Integration priority for `ACTIVE` entries; `—` for non-ACTIVE |

### Classification

| Class | Meaning | Action Implied |
|---|---|---|
| **INTEGRATED** | Structurally woven into skills / hooks / rules / standards. The lesson is enforced or canonized, not just documented. | None — verification only at periodic audits |
| **ACTIVE** | Load-bearing constraint NOT yet structurally woven. Recurrence risk if forgotten. | Queue for integration at priority P0/P1/P2 |
| **ADVISORY** | Wisdom / heuristic — useful to know, but no enforcement gate is needed (or feasible). | Periodic review only |
| **ARCHIVED** | Superseded by a later lesson, fixed by a structural change, or no longer applicable. | None — historical reference |

### Priority (ACTIVE only)

- **P0** — Integrate this iteration. Drives a recurring failure mode. Block on lack of integration.
- **P1** — Integrate this quarter. Load-bearing but the failure mode is currently contained.
- **P2** — Nice to have. No urgency, but a clean integration would be valuable.

### Re-classification cadence

LLs are re-scored at **major phase boundaries** (post-task or post-audit). Re-scoring is reader-determined, not protocol-mandated — Classification drift is itself a useful signal of structural integration progress (or stagnation). Modeled on `skills/audit-workspace/references/classification-schema.md` finding-classification + severity pattern.

| LL-20 | 2026-04-08 | proc | ACTIVE | P0 | Sycophantic agreement compounds with persistence gaps to produce recurring errors | When context is lost across compaction/sessions AND the default behavior is to agree with user framing rather than independently reason, errors compound: (1) incorrect assumptions survive unchallenged across sessions, (2) past lessons-learned are not proactively consulted before making decisions they should govern, (3) "hopeful" claims about capabilities go unvalidated because the user's framing isn't critically examined. **Concrete example**: LL-17 branch contamination — worktree best practice existed in KB but was not independently recommended because the session followed the user's implicit workflow without pushback. **Mitigation**: (1) Before agreeing with a user's approach, independently verify it against relevant KB and LL entries, (2) distinguish "this is what the user wants" vs "this is what the evidence supports" — flag divergence explicitly, (3) treat LL entries as hard constraints to re-read before any decision they govern. **Session 32 reinforcement**: Pivot 1 (drop LLs from bootstrap) and Pivot 2 (flat `notes/`) were both user-issued course corrections to my reflexive sycophantic adherence to existing structure. The v1 "shrink the file" plan was a sycophantic agreement to the original framing; v2 "fix the read, not the file" required independent reframing. | LL-28 | 2026-04-11 | proc | ACTIVE | P0 | State files capture plan-level state well but miss dialogue-level emergent content when sessions die mid-dialogue | LL-26 solved **tense staleness** (state files describing "pending" commits that already landed). It does NOT solve **emergence staleness** — discoveries, analyses, and decisions produced during execution *after* the last state-refresh checkpoint. **Concrete incident (Session 26 death via "Prompt is too long")**: Between Phase 5b state refresh (`726c50b`) and context exhaustion, the session also completed Phase B.5 push, Phase C.1 inventory, the LL-27 shadowing discovery (the session's most important finding), a 3-layer Identity/Behavioral/Selection architectural framing, and the user's unanswered HITL question. **None** of that reached `notes/` — Session 27 had to reconstruct it from the JSONL transcript. **Root cause**: `execute-task` Phase 5 and `close-session` Step 2 both write state at the end of a phase. Trigger events: (a) new architectural insight worth capturing, (b) HITL decision point reached (write pending question + options to state before asking), (c) investigation finding that contradicts a prior LL/DD, (d) discovery that implies a new LL entry. **Fallback recovery protocol**: When `Prompt is too long` strikes, new session must read `notes/` for plan state, read last ~20 turns of dying-session JSONL transcript for dialogue state, backfill emergent content as Session N+1 first action, draft any new LL entries, resume at the exact HITL question.
